## Hardware Architecture Summary: Gemmini

Gemmini is a configurable systolic-array accelerator for matrix multiplication and related tensor operations, generated by the Berkeley Gemmini hardware generator. This agent targets configurations with PE dimension other than 4 (typically `pe_dim=16` for the gemm/conv workloads).

### Decoupled Access/Execute Architecture

Gemmini is a decoupled access/execute architecture: memory-access and execute instructions happen concurrently in different regions of the hardware. The accelerator exposes three controllers:

- **ExecuteController** — handles `preload` and `compute` instructions on the systolic array.
- **LoadController** — handles `mvin` (DRAM → scratchpad/accumulator).
- **StoreController** — handles `mvout` (scratchpad/accumulator → DRAM).

Gemmini includes a reorder buffer (ROB) which detects hazards between instructions in different controllers. Each controller also handles its own dependencies and hazards internally. This means well-scheduled code can overlap loads, computes, and stores.

### Memory: Scratchpad and Accumulator

Gemmini's private memory is **row-addressed**: each row is `DIM` elements wide, where `DIM` is the number of PEs across the width of the systolic array. Elements are of type `inputType` in the scratchpad, and of type `accType` in the accumulator.

Every private Gemmini memory address is 32 bits long. The three most-significant bits are reserved:

- **Bit 31 (MSB)** is `0` for scratchpad addresses and `1` for accumulator addresses.
- **Bit 30** is ignored for scratchpad addresses or for accumulator reads. For accumulator writes, `0` overwrites the data at that address and `1` accumulates on top of it.
- **Bit 29** is ignored for scratchpad addresses or for accumulator writes. For accumulator reads, `0` reads scaled-down `inputType` data and `1` reads `accType` data. When bit 29 is `1`, no activation function or scaling is applied to the output.

### Systolic Array

Gemmini's compute is a `DIM × DIM` systolic array operating in either weight-stationary or output-stationary dataflow. A standard matmul step is:

1. `mvin` the A matrix tile and B matrix tile to the scratchpad.
2. `preload` the B tile onto the array, choosing the destination accumulator address.
3. `compute_preloaded` (first compute after preload) followed by zero or more `compute_accumulated` calls that reuse the same B in weight-stationary mode.
4. `mvout` the accumulated tile from the accumulator back to DRAM.

Because the scratchpad is row-addressed and tiles are at most `DIM` rows, large matrices must be tiled. Increasing tile size in the scratchpad reduces redundant DRAM traffic. Conditional `mvin` (`if`-gated loads) is a common idiom for skipping redundant loads when the same data is reused across outer loop iterations.

### Optimization Principles

- The single biggest performance lever is **reducing redundant data movement**: load each piece of A and B from DRAM as few times as possible by tiling and conditionally gating loads.
- **Decoupled execution** means loads, computes, and stores can overlap if dependencies are arranged correctly. Pipelining and double buffering pay off.
- **Spread data through the scratchpad** — repeated loads to the same address waste DMA bandwidth and serialize the LoadController.
- **Minimize loop overhead** — fewer outer loop iterations with larger tiles is usually better than many small tiles.
- The accumulator's accumulate-vs-overwrite bit lets contraction-dimension reductions happen without round-tripping to DRAM.
